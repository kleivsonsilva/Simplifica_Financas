Autenticação
Como usar o JWT
1
Faça login ou registro para obter o token JWT.
2
Inclua o token em todas as requisições protegidas no header:
Authorization: Bearer <seu_token_aqui>
3
O token expira em 24 horas. Após expirar, faça login novamente.
4
Ao fazer logout, o token é invalidado imediatamente via blacklist no Redis.
Rotas públicas (sem token)
POST /api/auth/registro
POST /api/auth/login
GET /health
Erro de autenticação — resposta padrão
HTTP 401 Unauthorized { "error": "Token não fornecido" // sem header Authorization "error": "Token inválido ou expirado" // token malformado "error": "Token revogado" // após logout }
Fluxos principais de comunicação
Fluxo de login e acesso
Browser → POST /api/auth/login → Gateway :8000 → proxy → Auth :5001 → PostgreSQL + Redis
← retorna JWT ←
Fluxo de criação de transação (síncrono + assíncrono)
POST /api/transacoes → Gateway → Transaction Service → MongoDB
↳ invalida cache de saldo no Redis
↳ publica evento "transacao_criada" no RabbitMQ
↳ Notification Service consome → salva alerta no Redis
↳ Goal Service consome → (processamento de metas)

Auth
Auth Service
interno: :5001 | externo via gateway: /api/auth/...

Responsável por registro, login, logout, JWT e gerenciamento de perfil. Banco: PostgreSQL + Redis.

POST /api/auth/registro Criar nova conta 🔓
Request Body (JSON)
{ "nome": "João Silva", // string, obrigatório "email": "joao@email.com", // string, obrigatório, único "senha": "minhasenha123", // string, obrigatório, mín. 6 chars "modo_interface": "simples" // "simples" | "avancado", padrão: "simples" }
Response 201 — Sucesso
{ "message": "Usuário criado com sucesso", "token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...", "usuario": { "id": "uuid-aqui", "nome": "João Silva", "email": "joao@email.com", "modo_interface": "simples", "data_criacao": "2026-05-28T14:00:00" } }
Respostas possíveis
201 Created 400 Campos obrigatórios faltando 400 Senha muito curta 409 Email já cadastrado 500 Erro interno
POST /api/auth/login Autenticar usuário 🔓
Request Body (JSON)
{ "email": "joao@email.com", // string, obrigatório "senha": "minhasenha123" // string, obrigatório }
Response 200 — Sucesso
{ "token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...", "usuario": { "id": "uuid-aqui", "nome": "João Silva", "email": "joao@email.com", "modo_interface": "avancado" } }
Respostas possíveis
200 OK 400 Campos obrigatórios 401 Credenciais inválidas 500 Erro interno
POST /api/auth/logout Encerrar sessão e revogar token 🔒
🔒 Requer Authorization: Bearer <token>
Request
Sem body. O token é extraído do header Authorization.
Response 200
{ "message": "Logout realizado com sucesso" }
Efeito
O token é adicionado à blacklist no Redis com TTL de 24h. Qualquer requisição subsequente com esse token retornará 401.
200 OK 401 Não autenticado

🔒 Requer Authorization: Bearer <token>

Notificações
Notification Service
interno: :5005 | externo via gateway: /api/notificacoes/...

Consome eventos do RabbitMQ e armazena alertas por usuário no Redis (máx. 50 notificações, LIFO).

Regras de geração de alertas (automáticas)
⚠ Alerta — Despesa > R$ 1.000,00
✅ Sucesso — Qualquer receita registrada
ℹ Info — Despesa normal (<= R$ 1.000,00)
🗑 Info — Transação excluída
GET /api/notificacoes Listar últimas 20 notificações 🔒
Response 200
{ "notificacoes": [ { "mensagem": "✅ Receita registrada: R$ 3500.00", "tipo": "sucesso", "data": "2026-05-28T14:00:00", "lida": false }, { "mensagem": "⚠️ Despesa alta registrada: R$ 1200.00 em Moradia", "tipo": "alerta", "data": "2026-05-27T10:30:00", "lida": false } ], "total": 2 }
200 OK 401 Não autenticado
DEL /api/notificacoes/limpar Limpar todas as notificações 🔒
Response 200
{ "message": "Notificações limpas" }
200 OK 401 Não autenticado
